# 📘 Cycle Detection in an Undirected Graph [BFS] [GFG [Link](https://www.geeksforgeeks.org/problems/detect-cycle-in-an-undirected-graph/1)] 


## Problem statement

Given an undirected graph with `V` vertices (numbered `0` to `V-1`) and a list of edges `edges` where each edge is a pair `[u, v]`, determine whether the graph contains a cycle. Return `true` if a cycle exists, otherwise `false`.

> **Assumptions to check** (often unstated):
>
> * Are vertices 0-indexed or 1-indexed? (This code assumes **0-indexed**.)
> * Are there parallel edges or self-loops? (They count as cycles for the usual definition.)
> * `edges` is a list of pairs like `[[0,1], [1,2], ...]`.



## Intuition 🧠

In an **undirected** graph, we can detect a cycle by traversing the graph (BFS or DFS) and ensuring that when we see an adjacent vertex that's already visited, it is *not* the parent in the traversal. If an already-visited neighbor is not the parent, we've found a cycle.

Why? 
>Because in undirected graphs, the immediate return-edge to parent is expected (that's the edge you came from). Any other visited neighbor indicates some different path connecting current node and that neighbor → cycle.




## Brute-force idea (not recommended)

Check for cycles by, for each node, performing a BFS/DFS but *recomputing neighbors by scanning the entire `edges` list* (what the original code does). This is the brute-force approach — it works, but is **inefficient** for large `E` and `V`. Complexity: **O(V * E)**.



## Optimal approach (clean, standard) ✅

1. **Build adjacency list** from `edges`: `adj[u].push_back(v); adj[v].push_back(u);`
2. **Use BFS (or DFS)** from every unvisited node. During BFS keep `(node, parent)`.

   * When exploring neighbors:

     * if neighbor not visited → mark visited and push `(neighbor, node)`
     * else if neighbor visited and neighbor != parent → cycle detected
3. **Return true** on first detection, else after all components, return false.

This approach runs in **O(V + E)** time and **O(V + E)** extra space for adjacency.

# BFS version (C++) 🛠️

```cpp
// Improved BFS-based cycle detection for undirected graph
#include <bits/stdc++.h>
using namespace std;

class Solution {
private:
    bool detectBFS(int src, const vector<vector<int>>& adj, vector<int>& vis) {
        queue<pair<int,int>> q;
        vis[src] = 1;
        q.push({src, -1});

        while (!q.empty()) {
            auto [node, parent] = q.front(); q.pop();
            for (int neighbor : adj[node]) {
                if (!vis[neighbor]) {
                    vis[neighbor] = 1;
                    q.push({neighbor, node});
                } else if (neighbor != parent) {
                    // visited and not the parent -> cycle
                    return true;
                }
            }
        }
        return false;
    }

public:
    bool isCycle(int V, const vector<vector<int>>& edges) {
        // build adjacency list
        vector<vector<int>> adj(V);
        for (const auto &e : edges) {
            if (e.size() < 2) continue; // defensive
            int u = e[0], v = e[1];
            // If input is 1-indexed, convert with u--, v--
            adj[u].push_back(v);
            adj[v].push_back(u);
        }

        vector<int> vis(V, 0);

        for (int i = 0; i < V; ++i) {
            if (!vis[i]) {
                if (detectBFS(i, adj, vis)) return true;
            }
        }
        return false;
    }
};
```



## Line-by-line explanation 🧩

* **Build adjacency list**
  `vector<vector<int>> adj(V);`
  For each edge `[u,v]`: `adj[u].push_back(v); adj[v].push_back(u);` — O(E)

* **vis array**
  `vector<int> vis(V, 0);` — 0 = unvisited, 1 = visited.

* **Traverse components**
  For each vertex `i` not visited, run BFS starting at `i`.

* **BFS with (node, parent)**
  `queue<pair<int,int>> q;` push `(src, -1)`. For each neighbor:

  * If not visited → mark visited, `q.push({neighbor, node})`.
  * If visited and `neighbor != parent` → we found a cycle.

* **Return**
  If any BFS returns true → overall `true`, otherwise `false`.



## Complexity analysis 📊

* **Time (improved)**: O(V + E)

  * Building adjacency: O(E)
  * Traversal: each vertex & edge processed once (neighbors scanned once total) → O(V + E)

* **Space**: O(V + E) for adjacency + O(V) for `vis` + O(V) worst-case for BFS queue / DFS recursion stack.

* **Time (original provided)**: O(V * E) worst-case — not ideal.



## Test cases ✅

Include corner cases and explanations.

1. **Simple cycle**

   * V = 3, edges = [[0,1],[1,2],[2,0]] → `true`

2. **Simple linear chain**

   * V = 4, edges = [[0,1],[1,2],[2,3]] → `false`

3. **Disconnected graph with a cycle**

   * V = 5, edges = [[0,1],[1,2],[2,0],[3,4]] → `true`

4. **Single node, no edges**

   * V = 1, edges = [] → `false`

5. **Self-loop**

   * V = 2, edges = [[0,0]] → `true` (self-loop is a cycle)

6. **Parallel edges (multi-edge)**

   * V = 2, edges = [[0,1],[0,1]] → `true` (two edges between same nodes create a 2-cycle in multigraph sense)

7. **1-indexed input (careful!)**

   * If edges are `[[1,2],[2,3]]` but V is number of vertices and nodes are 1..V, convert to 0-index by subtracting 1 or adjust code accordingly.



## The easiest way to remember this algorithm in interviews 🎯

> **“Build adjacency → traverse with parent-tracking → visited neighbor ≠ parent means cycle.”**

Say it like this:
>*"Neighbors visited before the parent = proof of a loop."* Short, sticky, and interview-friendly.



## A 5-step intuition trick so you never forget this problem (mnemonic: **P.A.T.H.S** ) 🧭

1. **P** — *Prepare adjacency* (convert edge list → adjacency list).
2. **A** — *All nodes* (visit every component).
3. **T** — *Track parent* for each node while traversing.
4. **H** — *Hit visited neighbor* → if it's not parent, you hit a cycle.
5. **S** — *Stop early* — return true as soon as you see that.

Recite PAT H S in your head during the interview.



## Tips, tricks & variations ✨

* **Use DFS** for slightly simpler recursive code; **BFS** is iterative and avoids recursion limits.
* **Watch indexing**: judge inputs sometimes use 1-indexing.
* **Multi-edges & self-loops**: decide if they count as cycles. Usually they do.
* **Directed graphs**: totally different. Use DFS + recursion stack or Kahn’s algorithm (topological sort) — don't reuse this parent-check trick.
* **Large sparse graphs**: adjacency lists are mandatory — adjacency matrix would be wasteful.
* **Non-integer vertex labels**: map them to integers with unordered_map if necessary.
* **Stack overflow**: for DFS on deep graphs, iterative DFS or BFS is safer.
* **Parallel edges detection**: adjacency list detects them naturally as cycle (two edges between u and v).



## Variations & extensions 🔄

* **Detect cycle in directed graph** — use [DFS with recursion stack](05.CycleInUndirectedII.md) or color marking (white/gray/black).
* **Return cycle nodes** — when detected, reconstruct path using parent array.
* **Count cycles** — harder, need careful handling to avoid double counting.
* **Union-Find (Disjoint Set)** can detect cycles in an undirected graph too:

  * For each edge `u-v`, if `find(u) == find(v)`, cycle; else `union(u,v)`.
  * That’s **very common** and runs in near O(α(N)) per op. Good alternative for "online" checks.



## FAQ (common interview questions) ❓

**Q: Does parent-check method work for parallel edges?** <br>
A: Yes — two parallel edges between `u` and `v` will either be seen as a cycle earlier (depends on the order) but they are a cycle in a multigraph sense.

**Q: What about self-loops?** <br>
A: Self-loop `[u,u]` means `u` has adjacent `u` which is already visited — code will detect it as a cycle (good).

**Q: Should I use BFS or DFS?** <br>
A: Either is fine. DFS is slightly more concise; BFS is iterative and safe if recursion depth is a concern.

**Q: When should I use Union-Find?** <br>
A: When you're only processing edges and want a very clean cycle check (and/or the graph is given as a stream of edges). It's also commonly accepted in interviews.

**Q: What if vertices are labeled 1..V?** <br>
A: Convert to 0-based indexing: `u--, v--` when building adjacency.



## Quick comparison: BFS/DFS vs Union-Find

* **BFS/DFS**

  * Works naturally for connectivity + cycle detection.
  * O(V + E) time, O(V + E) space.
  * Can easily be extended to reconstruct cycle path.

* **Union-Find**

  * O(E α(V)) time, O(V) space.
  * Simpler if you only need to know yes/no for cycles while processing edges.
  * Not suited for *directed* graphs.
