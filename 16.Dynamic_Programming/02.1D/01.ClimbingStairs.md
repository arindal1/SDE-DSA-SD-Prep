
# 🧗‍♂️ Climbing Stairs with Memoization [Leetcode #[70](https://leetcode.com/problems/climbing-stairs/)]

This is a classic **Dynamic Programming** problem dressed up as a simple staircase question. The code you shared is a **top-down DP** solution using **memoization**.



## ✅ Problem Statement

You are standing at the bottom of a staircase with `n` steps.

At each move, you can climb:

* **1 step**
* **2 steps**

Return the **number of distinct ways** to reach the top.



## 🧠 Core Idea

This problem is not really about stairs.

It is about **counting ways** where each position depends on earlier positions.

For step `n`, the last move must have come from:

* step `n - 1` using a `1-step`
* step `n - 2` using a `2-step`

So the recurrence becomes:

```text
ways(n) = ways(n - 1) + ways(n - 2)
```

That is basically Fibonacci with a different story wrapper. Very Marvel origin-story energy.



## 🧠 Intuition

Think of the staircase like a path.

To reach step `n`, your **last jump** must be one of two things:

* a jump from `n - 1`
* a jump from `n - 2`

So instead of directly asking:

> “How do I get to `n`?”

we ask:

> “How many ways are there to get to the previous steps?”

That recursive breakdown is what makes DP work here.

## 🧩 CPP Code

```cpp
class Solution {
public:
    int solve(int n, vector<int>& dp) {
        if (n == 0)
            return 1;
        if (n < 0)
            return 0;

        if (dp[n] != -1)
            return dp[n];
        return dp[n] = solve(n - 1, dp) + solve(n - 2, dp);
    }

    int climbStairs(int n) {
        vector<int> dp(n + 1, -1);
        return solve(n, dp);
    }
};
```

## 🔍 Brute Force Approach

### Idea

Try all possible ways:

* take `1` step
* take `2` steps

Repeat until you hit the top.

### Recursive brute force

```cpp
int climbStairs(int n) {
    if (n == 0) return 1;
    if (n < 0) return 0;
    return climbStairs(n - 1) + climbStairs(n - 2);
}
```

### Why it is slow

This recomputes the same states again and again.

Example:

* `climbStairs(5)` calls `climbStairs(4)` and `climbStairs(3)`
* `climbStairs(4)` again calls `climbStairs(3)` and `climbStairs(2)`

So `climbStairs(3)` gets calculated multiple times.

### Time complexity of brute force

`O(2^n)`

That is too expensive for large `n`.



## 🚀 Optimal Approach: Memoization

Memoization means:

> **Solve once, store once, reuse forever**



### State Definition

Let:

```text
dp[n] = number of ways to reach step n
```



### Transition

To compute `dp[n]`:

```text
dp[n] = dp[n - 1] + dp[n - 2]
```

because the last move can come from:

* `n - 1` by taking 1 step
* `n - 2` by taking 2 steps



### Base Cases

```text
dp[0] = 1
dp[n < 0] = 0
```

In the recursive version, these are handled directly.



## Memoized Recursive Flow

When `solve(n)` is called:

1. If `n == 0`, return `1`
2. If `n < 0`, return `0`
3. If `dp[n]` already contains an answer, return it
4. Otherwise compute:

   ```cpp
   solve(n - 1) + solve(n - 2)
   ```
5. Store the result in `dp[n]`
6. Return it

This prevents repeated work.



## 🧮 Walkthrough Example

Let `n = 4`.

We want `ways(4)`.

### Expansion:

```text
ways(4) = ways(3) + ways(2)
ways(3) = ways(2) + ways(1)
ways(2) = ways(1) + ways(0)
```

Base values:

```text
ways(1) = 1
ways(0) = 1
```

So:

* `ways(2) = 1 + 1 = 2`
* `ways(3) = 2 + 1 = 3`
* `ways(4) = 3 + 2 = 5`

### Final answer:

```text
5
```

The 5 ways are:

1. `1 + 1 + 1 + 1`
2. `1 + 1 + 2`
3. `1 + 2 + 1`
4. `2 + 1 + 1`
5. `2 + 2`



## 🧠 Full Algorithm

### Memoization Algorithm

1. Define `dp[n]` as the number of ways to climb to step `n`
2. Initialize all `dp` values to `-1`
3. Use recursion:

   * if `n == 0`, return `1`
   * if `n < 0`, return `0`
   * if `dp[n]` is already computed, return it
   * otherwise compute:

     ```cpp
     solve(n - 1) + solve(n - 2)
     ```
4. Store the result in `dp[n]`
5. Return `solve(n)`

## 🔧 Full Code with Clearer Comments

```cpp
class Solution {
public:
    int solve(int n, vector<int>& dp) {
        // Base case: one valid way to stay at step 0
        if (n == 0) return 1;

        // Invalid path
        if (n < 0) return 0;

        // Already computed
        if (dp[n] != -1) return dp[n];

        // Number of ways = ways from one step below + two steps below
        return dp[n] = solve(n - 1, dp) + solve(n - 2, dp);
    }

    int climbStairs(int n) {
        vector<int> dp(n + 1, -1);
        return solve(n, dp);
    }
};
```

For step `n`, the answer is:

* from `n - 1`

### 2. Translate that into a formula

```text
ways(n) = ways(n - 1) + ways(n - 2)
```



### 3. Add base cases

```text
ways(0) = 1
```

The same subproblems keep showing up again and again.

### 5. Cache the answer

## 💡 Tips and Tricks

### Tip 1: `n == 0` returning `1` is intentional

This is not a bug. It represents one valid empty path.



### Tip 2: Do not confuse “number of steps” with “number of ways”

The state is not the stairs themselves. The state is the **count of ways to reach a step**.



### Tip 3: If recursion is clear, memoization is the easiest first solution

This is often the smoothest way to write DP in interviews.



### Tip 4: You can optimize this further

This recurrence only depends on the last two values, so it can also be solved using **space optimization** with two variables.



### Tip 5: Recognize the Fibonacci pattern

This problem is structurally Fibonacci-like.

That is a very useful mental shortcut.



## 🔄 Variation Ideas

This problem has many close cousins.

### 1. Climb stairs with 1, 2, or 3 steps

Now the recurrence becomes:

```text
ways(n) = ways(n - 1) + ways(n - 2) + ways(n - 3)
```



### 2. Min cost climbing stairs

Instead of counting ways, you minimize total cost.



### 3. Variable jump stair problem

At each step, you can jump up to `arr[i]` steps.



### 4. Restricted steps

Some stairs may be blocked. Then you skip invalid states.



## 🚀 Space-Optimized Version

Since only the previous two results are needed, the array is not strictly necessary.

```cpp
class Solution {
public:
    int climbStairs(int n) {
        if (n == 0) return 1;
        if (n == 1) return 1;

        int prev2 = 1; // ways(0)
        int prev1 = 1; // ways(1)

        for (int i = 2; i <= n; i++) {
            int current = prev1 + prev2;
            prev2 = prev1;
            prev1 = current;
        }

        return prev1;
    }
};
```

### Complexity

* **Time:** `O(n)`
* **Space:** `O(1)`

This is the cleaner “final boss” version once you understand the DP logic.



## ❓ FAQs

### Q1. Why do we return `1` for `n == 0`?

Because there is exactly one way to do nothing and stay where you are.
This is a standard DP/combinatorics convention.



### Q2. Why do we return `0` for `n < 0`?

Because negative steps are invalid.
That path should not contribute to the count.



### Q3. Is this problem actually Fibonacci?

Yes, very much so.
The recurrence is the same structure:

```text
F(n) = F(n-1) + F(n-2)
```

The interpretation is different, but the math is nearly identical.



### Q4. Why use memoization instead of plain recursion?

Plain recursion repeats the same subproblems many times.

Memoization stores answers and avoids recomputation.



### Q5. Can this be solved iteratively?

Yes. That is the tabulation or space-optimized version.



### Q6. Is the memoized solution acceptable in interviews?

Absolutely.
It is a correct DP solution and often easier to explain than tabulation at first.
