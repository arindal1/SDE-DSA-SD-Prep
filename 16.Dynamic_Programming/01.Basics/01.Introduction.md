# 📘 Dynamic Programming (DP): Memoization, Tabulation, and Space Optimization


## 1) What is Dynamic Programming? 🧠

**Dynamic Programming** is a technique used to solve problems by breaking them into smaller subproblems, solving each subproblem once, and reusing those results.

It is especially useful when:

* the problem has **overlapping subproblems**
* the problem has **optimal substructure**

### Overlapping subproblems

The same subproblem is computed many times in a naive recursive solution.

Example:

```text
fib(5)
  -> fib(4) + fib(3)
  -> fib(3) + fib(2) + fib(2) + fib(1)
```

Here, `fib(3)` and `fib(2)` get repeated.

### Optimal substructure

The best solution to the whole problem can be built from the best solutions to smaller parts.

Example:

* shortest path
* maximum profit
* minimum cost
* longest subsequence



## 2) Why DP is needed? ⚡

Many recursive solutions are logically correct, but they are too slow because they repeat work.

### Naive recursion example

Fibonacci:

```cpp
int fib(int n) {
    if (n <= 1) return n;
    return fib(n - 1) + fib(n - 2);
}
```

### Problem

This has **exponential time complexity** because it recomputes the same values again and again.

### DP fixes this by:

* storing computed answers
* reusing them when needed



## 3) Core idea of DP 🎯

When solving a DP problem, think in this order:

1. **State**
   What does `dp[i]` or `dp[i][j]` mean?

2. **Transition**
   How do you compute the current state from previous states?

3. **Base case**
   What are the smallest answers you already know?

4. **Order of computation**
   In what order should states be filled?

5. **Answer extraction**
   Which state contains the final answer?



## 4) Memoization (Top-Down DP) 💾

Memoization is **recursion + caching**.

You start from the main problem, recursively go down to smaller subproblems, and store answers so they are not recomputed.

### Idea

* Solve the problem recursively
* Before computing a state, check if it is already stored
* If yes, return it immediately
* If not, compute and store it



### Memoization example: Fibonacci

```cpp
#include <bits/stdc++.h>
using namespace std;

int solve(int n, vector<int>& dp) {
    if (n <= 1) return n;

    if (dp[n] != -1) return dp[n];

    return dp[n] = solve(n - 1, dp) + solve(n - 2, dp);
}

int fib(int n) {
    vector<int> dp(n + 1, -1);
    return solve(n, dp);
}
```



### How it works

For `fib(5)`:

* compute `fib(5)`
* need `fib(4)` and `fib(3)`
* `fib(4)` needs `fib(3)` and `fib(2)`
* when `fib(3)` is computed once, store it
* later reuse it instead of recalculating



### Time and space complexity

* **Time:** `O(n)`
* **Space:** `O(n)` for dp array
* **Extra recursion stack:** `O(n)`

So total auxiliary stack usage is still significant.



### When memoization is useful

Use memoization when:

* the recursive relation is easy to write
* the state space is not too large
* you want a natural top-down solution

It is often the easiest way to convert recursion into DP.



### Advantages of memoization ✅

* Easy to derive from recursion
* Good for problems where not all states are needed
* Often easier to understand initially



### Disadvantages of memoization ❌

* Recursion stack can still be large
* Slight overhead from recursive calls
* Sometimes harder to optimize further



## 5) Tabulation (Bottom-Up DP) 📋

Tabulation is the opposite style. Instead of starting from the final answer and going down recursively, you start from the base cases and build upward.

### Idea

* Create a DP table
* Fill it iteratively
* Each state is computed using previously filled states



### Tabulation example: Fibonacci

```cpp
#include <bits/stdc++.h>
using namespace std;

int fib(int n) {
    if (n <= 1) return n;

    vector<int> dp(n + 1, 0);
    dp[1] = 1;

    for (int i = 2; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }

    return dp[n];
}
```



### How it works

We already know:

* `dp[0] = 0`
* `dp[1] = 1`

Then:

* `dp[2] = dp[1] + dp[0]`
* `dp[3] = dp[2] + dp[1]`
* `dp[4] = dp[3] + dp[2]`

and so on.



### Time and space complexity

* **Time:** `O(n)`
* **Space:** `O(n)`

This removes recursion overhead, which is often a practical improvement.



### Advantages of tabulation ✅

* No recursion stack
* Usually faster in practice
* More control over iteration order
* Easier to optimize later



### Disadvantages of tabulation ❌

* Sometimes harder to derive initially
* You must carefully decide iteration order
* You often compute all states, even unnecessary ones



## 6) Memoization vs Tabulation ⚔️

| Aspect             | Memoization               | Tabulation                 |
| - | - | -- |
| Style              | Top-down                  | Bottom-up                  |
| Approach           | Recursion + cache         | Iterative table filling    |
| Base case handling | Inside recursive function | Explicit initialization    |
| Stack usage        | Uses recursion stack      | No recursion stack         |
| Ease for beginners | Often easier              | Sometimes harder initially |
| Performance        | Good                      | Usually slightly better    |



### Simple intuition

* **Memoization** feels like: “Let me ask the question and remember the answers.”
* **Tabulation** feels like: “Let me build the answers step by step.”

Both solve the same recurrence. The difference is in the direction of computation.



## 7) How to recognize a DP problem 🔍

A problem is likely DP if:

* it asks for **minimum**, **maximum**, **count**, or **boolean possibility**
* the same subproblem appears again and again
* the problem can be broken into smaller states
* the solution depends on previous decisions

### Common DP keywords

* minimum cost
* maximum profit
* number of ways
* can we reach
* longest / shortest
* optimal sequence



## 8) How to design a DP solution 🧩

This is the most important part.

### Step 1: Define the state

Ask: what does `dp[i]` mean?

Examples:

* `dp[i] = minimum cost to reach i`
* `dp[i] = number of ways to reach i`
* `dp[i][j] = answer using first i items and capacity j`

A good state should be:

* precise
* small enough to store
* enough to derive the answer



### Step 2: Write the transition

Ask: how do I compute `dp[i]` from smaller values?

Example:

```cpp
dp[i] = dp[i - 1] + dp[i - 2];
```

Or:

```cpp
dp[i] = min(dp[i - 1], dp[i - 2]) + cost[i];
```



### Step 3: Identify the base cases

Without base cases, DP cannot start.

Example:

```cpp
dp[0] = 0;
dp[1] = 1;
```



### Step 4: Choose the direction

If the current state depends on previous states, fill from left to right or bottom to top.

If using memoization, recursion handles the order naturally.



### Step 5: Return the final answer

Often it is:

* `dp[n]`
* `dp[n][m]`
* `max(dp[i])`
* `min(dp[i])`



## 9) A complete example: Climbing Stairs 🪜

Problem:
You can climb 1 or 2 steps at a time. How many ways are there to reach step `n`?



### Recurrence

To reach step `n`, you can come from:

* step `n - 1`
* step `n - 2`

So:

```text
ways(n) = ways(n - 1) + ways(n - 2)
```

Base cases:

```text
ways(0) = 1
ways(1) = 1
```



### Memoization solution

```cpp
#include <bits/stdc++.h>
using namespace std;

int solve(int n, vector<int>& dp) {
    if (n == 0 || n == 1) return 1;

    if (dp[n] != -1) return dp[n];

    return dp[n] = solve(n - 1, dp) + solve(n - 2, dp);
}

int climbStairs(int n) {
    vector<int> dp(n + 1, -1);
    return solve(n, dp);
}
```



### Tabulation solution

```cpp
#include <bits/stdc++.h>
using namespace std;

int climbStairs(int n) {
    vector<int> dp(n + 1, 0);
    dp[0] = 1;
    dp[1] = 1;

    for (int i = 2; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }

    return dp[n];
}
```



### Space optimized solution

Notice that `dp[i]` only depends on the previous two values.

So we do not need the whole array.

```cpp
#include <bits/stdc++.h>
using namespace std;

int climbStairs(int n) {
    if (n == 0 || n == 1) return 1;

    int prev2 = 1; // dp[0]
    int prev1 = 1; // dp[1]

    for (int i = 2; i <= n; i++) {
        int current = prev1 + prev2;
        prev2 = prev1;
        prev1 = current;
    }

    return prev1;
}
```



## 10) Space Optimization for Tabulation 🚀

Space optimization is a refinement of tabulation.

### Key observation:

If a DP state only depends on a small number of previous states, then you do not need the full array.

#### Example

In Fibonacci:

```text
dp[i] depends only on dp[i - 1] and dp[i - 2]
```

So:

* full array is unnecessary
* only two variables are enough



### General rule:

If:

```text
dp[i] depends on last 1 state -> use 1 variable
dp[i] depends on last 2 states -> use 2 variables
dp[i] depends on last k states -> use k variables
```



### Another example: Minimum cost stairs -

Suppose:

```cpp
dp[i] = cost[i] + min(dp[i - 1], dp[i - 2])
```

Again, only the previous two states matter.

### Space optimized version

```cpp
#include <bits/stdc++.h>
using namespace std;

int minCostClimbingStairs(vector<int>& cost) {
    int n = cost.size();
    if (n == 0) return 0;
    if (n == 1) return cost[0];

    int prev2 = cost[0];
    int prev1 = cost[1];

    for (int i = 2; i < n; i++) {
        int current = cost[i] + min(prev1, prev2);
        prev2 = prev1;
        prev1 = current;
    }

    return min(prev1, prev2);
}
```



### Benefits of space optimization ✅

* lower memory usage
* often faster due to better cache locality
* cleaner when only a few previous states are needed



### Caution ⚠️

Do **not** optimize space blindly.

Some DP problems need:

* previous row and current row
* full history
* reconstruction of path/choices

In such cases, full storage may still be required.



## 11) Common DP patterns 📚

Here are some very common shapes of DP problems.

### 1D DP:

State depends on a single index.

Examples:

* Fibonacci
* Climbing stairs
* House robber
* Minimum cost path on line

Typical form:

```cpp
dp[i] = function(dp[i - 1], dp[i - 2], ...)
```



### 2D DP:

State depends on two indices.

Examples:

* Longest common subsequence
* 0/1 Knapsack
* Grid path problems

Typical form:

```cpp
dp[i][j] = function(dp[i - 1][j], dp[i][j - 1], dp[i - 1][j - 1])
```



### Subsequence DP

Common in string problems and knapsack style problems.

Usually asks:

* take or skip
* match or not match



### Interval DP

State represents answers on subarrays or ranges.

Example:

* matrix chain multiplication
* palindrome partitioning



## 12) How to write DP in interviews or contests 📝

A clean way to explain your solution is:

### 1. State definition

> “Let `dp[i]` represent ...”

### 2. Transition

> “From state `i`, we can come from ...”

### 3. Base cases

> “Initially ...”

### 4. Order of filling

> “I will fill the table from ... to ...”

### 5. Complexity

> “Time is ..., space is ...”

This structure makes your solution feel clear and intentional.



## 13) Common mistakes to avoid 🚫

### 1. Wrong state definition

If the state is vague, the recurrence becomes messy.

Bad:

```text
dp[i] = best answer till i
```

Better:

```text
dp[i] = minimum cost to reach step i
```

Be precise.



### 2. Forgetting base cases

A DP solution without base cases is incomplete.



### 3. Incorrect transition

Make sure the transition matches the problem conditions exactly.



### 4. Wrong iteration order

In tabulation, order matters a lot.

If `dp[i]` depends on `dp[i - 1]`, then `dp[i - 1]` must already be computed.



### 5. Space optimizing too early

First get the correct DP.
Then optimize space only if the recurrence supports it.



## 14) Mental model for DP 🌱

A useful way to think about DP:

* recursion is the **idea**
* memoization is the **idea with memory**
* tabulation is the **idea built bottom-up**
* space optimization is **tabulation compressed**

So the pipeline is often:

```text
Recursive thinking
    -> Memoization
    -> Tabulation
    -> Space optimization
```



## 15) A small cheat sheet 🧾

### Memoization

Use when:

* recursion is natural
* repeated states exist
* you want a quick DP conversion

### Tabulation

Use when:

* you can determine a clear order
* recursion stack is a concern
* you want a more performant iterative approach

### Space optimization

Use when:

* current state depends only on a few previous states
* full DP array is not necessary



## 16) Final summary ✨

Dynamic Programming is about **not solving the same problem twice**.

* **Memoization** stores answers during recursion.
* **Tabulation** builds answers iteratively from the base cases.
* **Space optimization** compresses the DP table when only a few previous states are needed.

The real skill in DP is not memorizing formulas. It is learning how to define:

* the state
* the transition
* the base cases
* the computation order
